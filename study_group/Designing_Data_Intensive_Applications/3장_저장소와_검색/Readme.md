append-only db에 insert는 매우 빠른 작업

but, 항상 가져올 때가 문제임. 색인이라는 메타데이터를 유지해서 데이터 위치 찾기에 도움을 받자. 

그럼 쓰기 속도에 지장을 주지만, 자동으로 생성되는 색인을 구성하거나, 수동으로 생성을 시켜줘야 함. 

insert하다가 파일이 너무 커지면 segment로 로그를 나눠서 저장. 지금 파일 닫고, 다음 파일에 쓰기. 

나중에 compaction 해서, 중복된 키값은 버리고 (overwritten) 각 키 별로 최신 갱신 값만 유지.

#### 해시 색인

램위에 모든 해시 테이블을 올려놔야 빠르긴 하겠다. 

키 값이 자주 갱신되지만, 다만 고유 키가 많지 않은 경우. (키는 url이고 비디오 조회수 등을 저장하는 방식)

세그먼트 병합과 추가가 순차적 쓰기라 빠르고 (spatial localization), 복구가 간단하다. 동시성도 간단 

디스크에 저장하면 속도가 안나오니 메모리에 저장되야하는게 일이다. 


#### SS테이블과 LSM 트리 

- Sorted String Table (SS테이블)
  - 키 값 쌍을 키로 정렬한채로 저장되게 한다.
  - 각 세그먼트는 정렬이보장됨. RB 트리나, AVL 트리처럼 정렬보장 삽입 사용 (memtable 인메모리 테이블 사용 -> 이게 디스크에 써지면 하나의 segment됨)
  - 읽기 요청은 memtable로 먼저, 그다음 최신 세그먼트.
  - 백그라운드에서 세그먼트끼리 merge sort하듯이 합침

Cassandra,HBase 의 엔진구조야. 


Log-Structured Mergre-Tree (LSM 트리)

Lucene에서도 용어사전 저장하려고 유사한 방법 

문제점: 존재하지 않는 키 찾기 but 블룸 필터 (Bloom filter) 사용해서 극복함. 

블룸 필터: 어떤 키가 들어왔을 때, 그 키에 대해 같은 해시 함수들을 돌려서 가리킨 모든 비트 중 하나라도 0이면, 그 키는 절대 넣은 적이 없는 것이다. 없는 키에 의해 그 비트가 1이 될 방법이 없기 때문이다
  - 최악의 경우를 줄여주는 방법


#### B 트리

가장 유명한 색인 방식 

- 고정 크기 블록과 페이지로 나누고, 한번에 하나의 페이지에 읽기 또는 쓰기.
- 항상 자신의 위치를 찾아서 leaf 페이지에 쓰려고 해.
- WAL (Write Ahead Logging) 해서 B 트리 복원 할 수 있게 해야 한다. (보통 n key 가진 b tree에서 n개 key가 될 때)
- latch (가벼운 lock) 로 동시성 제어 


LSM: 같은내용 쓰는 게 많음 (write amplification) 다만 b트리보다 많지는 않음. (WAL하고, 페이지 전체를 다시 쓸 일은 없으니까)
  - 그래도, 순차쓰기라서 + 파편화 없애서 page에 남는공간없이 잘쓴다.
  - 만약 컴팩션이 느리다면, 읽기가 느려진다( 디스크에 잔존하는 파일이 많아져서 )


#### 보조색인

key가 가리키는 value는 실제 값이거나, 다른 row에 대한 reference. reference를 저장하는 곳이 heap file이다. 

clustered index는 색인안에 바로 색인된row를 저장하는 방식으로 저장한다. 보조 색인이 항상 기본키를 참조하면, 기본키 색인을 사용할 수 있다.
