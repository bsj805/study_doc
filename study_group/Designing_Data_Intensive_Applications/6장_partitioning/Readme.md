## 키-값 데이터 파티셔닝

노드 10개중 한개에만 부하가 몰리면 핫스팟 노드가 돼. (skewed 파티션임)

skew 회피는 노드 무작위 할당으로 해결할 수야 있지만, 특정 레코드가 어떤 노드에 저장되었는지 모르게 돼

### 키 범위를 파티셔닝

a~b c~e ...

특정 접근 패턴이 핫스팟 유발할  수 있다. 

### 키 해시값 파티셔닝

좋은 해시 함수가 쏠린 데이터를 균일하게 분산되게 함 

java의 `Object.hashCode()` 는 메모리주소 기반의 implementation이라 jvm instance 별로 달라져 

String.hashCode나 Integer, Long 의 hashCode() 함수는 동일한 결과를 낸다. 

## 키-값 데이터 파티셔닝 + 보조 색인 

보조 색인은 PK찾는 방식이 아니라, 특정 값에 대해 찾으면 여러 row가 나올 수 있는 상황 

### 문서기준 보조 색인 파티셔닝 (Scatter and Gather)

문서 PK보고 파티셔닝하고, 노드별로 보조 색인 유지. 모든 노드로 질의 보내야

### 용어기준 보조 색인 파티셔닝 (term-partitioned)

보조색인으로 지정된 칼럼에 대해서도 별도 색인을 두어서, 보조색인의 해시값 등으로 보조색인으로 pk를 먼저 조회하면 그 다음에 pk들을 가져온다.   

모든 파티션 아닌 특정 파티션만 읽으면 되어서 효율적이나, 쓰기가 느리고 복잡하다. 보조색인을 갱신하는게 비동기 갱신이라서.

## 파티션 리밸런싱 (rebalancing)

해시값 기준으로 `mod N` 했으면 노드 개수 N이 바뀌었을때 엄청 많은 데이터가 옮겨져야 한다. 

### 파티션 고정
파티션을 더 상세하게 쪼개고, 노드가 추가되면 해당 파티션들 몇개씩을 옮겨주면 심플하다. 총 파티션 개수는 변경되지 않는다. 

### 동적 파티셔닝
키 범위 파티셔닝을 쓰면 고정 파티션 개수에서는 한 버킷이 비어버릴 위험성 있다. 그래서 파티션 크기가 일정 값 넘어가면 파티션을 쪼갠다.

반대로 합쳐지기도 하고. 


### 파티션 리밸런싱시 요청 라우팅 

어떤 노드에 내가 원하는 파티션이 있는가? (Service Discovery problem)

- 아무노드에나 접속해서 원하는 파티션 찾기
- 라우팅 계층이 원하는 노드로 보내주기 
- 클라이언트가 어떤 파티션이 어떤 노드인지 알게하고 직접보내기 

zookeeper는 카프카 클러스터 만들때 봤었는데 이런용도였구나

Zookeeper라는 클러스터 메타데이터 추적하는 코디네이션 서비스 사용한다. 
- 각 노드는 주키퍼에 자신을 등록하고
- 주키퍼는 파티션과 노드 사이에 신뢰성 있는 할당 정보를 관리 
- 라우팅 계층이나 클라이언트가 주키퍼 정보 구독할 수 있다. 그래서 주키퍼가 파티션 소유자가 바뀌거나, 노드 추가 삭제 이벤트를 알린다.
