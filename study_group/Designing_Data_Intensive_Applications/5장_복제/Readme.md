## 리더와 팔로워 

leader-based replication (master/slave replication)

- 클라이언트는 리더에게 요청, 리더는 로컬에 저장후, replication log나 change stream의 일부로 팔로워(slave)에게 전송 
- 읽기는 리더, 팔로워 모두에 가능 

현실적으로 모든 팔로워를 동기식으로 할 순 없고, 팔로워 하나는 동기식, 그 외는 비동기식. (semi-synchronous) 

그치만 보통 완전 비동기식으로 함. (많은 팔로워 or 지리적 분산) 


### 체인복제

대안으로 나온 체인 복제는  


Head  →  Replica1  →  Replica2  →  Tail
```
Head : 모든 쓰기 요청이 처음 도착

중간 노드들 : 순차적으로 복제

Tail : 최종 커밋 지점 + 읽기 담당
```

Tail 노드가 fail하면 Replica2가 Tail노드가 된다. 아니면 그냥 순차에서 빠진다.

Tail까지 반영되야 성공이라 동기적 복제인데, 왜 좋은걸까?

```
                   → Replica1
                 ↗
Client → Primary → Replica2
                 ↘
                   → Replica3
```

기존 Primary-replica 식은 primary가 모든 복제 ack를 직접 관리했어야 하기 때문에 
- 네트워크 fan-out 비용 증가
- 읽기 시 어디가 최신인지 판단 필요 
- primary가 병목임


체인 복제시 항상 일관성있게 Tail은 무조건 읽기를 보장. Head는 항상 쓰기를 보장. 

읽기, 쓰기에 대해 트래픽이 분산된다. (Primary에게 최신 데이터 질의하지 않아도 됨)

latency는 primary-replica가 좋지만, throughput은 chain방식이 좋다. (네트워크 병목이 줄어들기 때문에 + 락도 걸리지 않기 때문에)

```
Microsoft 연구에서 밝혔던 장점:

쓰기 순서 보장 쉬움

락 거의 없음

일관성 증명 단순

구현 안정성 높음

대규모 Key-Value 저장소에서
동시 업데이트가 많을 때 효과가 큼
```

## 새로운 팔로워 설정 

leader를 failover 할 때 

- 비동기식 복제 사용시 일반적인 복구방법은, 복제되지 않은 데이터는 폐기처리 하는 것 
  - out-of-date mysql follower가 리더로 승격된 사례가 있기 때문
- split brain 문제가 생길 수도 있음 (두 리더가 생김)
- 리더가 실제 장애상황이 아닌데 장애로 판단하면? 


## 복제 로그 구현 

### 가장 간단하게, 모든 쓰기 요청을 그대로 

INSERT, UPDATE, DELETE를 그대로 전달 문제점 
- NOW(), RAND() 같은거 -> 구문 기록시 리더가 고정값 넣어야 함 
- UPDATE WHERE 문 경우에는 순서대로 실행되야함 

1번같은 문제 때문에 에지 케이스가 많아서 다른 복제 방법을 사용한다 .

### 쓰기 전 로그 배송 
저장소 엔진이 디스크 상에 데이터 저장할때 WAL (Write ahead log) 에 쓴다. 

동일한 로그를 전송하면 동일한 데이터 구조 복제본이 만들어질것이다. 

문제점
- 로그가 저수준 데이터다. 저장소 엔진에 dependency가 생겨
- 엔진 버전 업 해야 하면 중단 생김 

### 논리적(row based) 로그 복제

저장소 엔진 로그가 아니라, logical log를 사용하자. 

대개는 DB 테이블에 쓰기로 되어 있는 row를 의미한다.

PK만 기록해도 되고, PK 없다면 이전의 모든 값을 이용해 로깅한다. 

### trigger based replication

보통 DB 시스템에 의해 구현되지만, DB의 서브셋만 복제하거나, 일부만 복제 하고 싶은 경우엔 어플리케이션에 의한 복제 이뤄져야함 

DB시스템이 trigger라는 기능을 제공. custom 어플리케이션 코드 등록 가능함. 

## 복제 지연 문제 
확장성 , 지연 시간 도 줄이는게 목표인데, 사용자가 근처 복제 서버에서 읽으려고 할 때 팔로워가 뒤쳐진 상태라면? 

복제지연 사례별 해결방법

### 자신이 쓴 내용 읽기 
READ_COMMITTED, 즉 자신이 썼다면 그 데이터 까진 볼 수 있게 하는 것

- 사용자가 수정한 내용 읽을 때는 리더에서 읽는다 (ex. SNS의 사용자 소유의 프로필 수정은 항상 리더에서만 발생하고 리더에서만 읽게 규정)
- 마지막 갱신 후 1분 동안은 리더에서 모든 읽기 수행 (팔로워들 중 리더보다 1분 이상 늦은 모든 팔로워에 대한 질의 금지 등)
- 클라이언트에서 자신의 쓰기 타임스탬프를 기억해서, 읽기 요청이 온 사용자의 타임스탬프보다 동기화된 타임스탬프 느리다면 질의 대기 시키기. 

cross-device 에서의 read consistency도 문제될 수도 있다. 이 경우는 3번같은 방법이 불가능하다. 중앙에서 읽기 요청이 온 시간을 관리해줘야 한다. 

### Monotonic Read 
monotonic read가 보장 안되면, 이전에 보였던 데이터가 없어지는 현상이 보일 수도 있다. 
특정 사용자의 read request를 사용자의 ID 해시 기반으로 복제 서버를 결정짓게 해서, 항상 동일한 서버로 라우팅되게. 

### consistent prefix read

일관된 순서로 읽어야 한다. 일반적으로 DB들이 같은 순서로 쓰기를 적용하지 않기 때문. 

인과성 있는 쓰기를 동일 파티션에 기록되게 할 수는 있으나 효율적이진 않음.

## New Solution: 다중 리더 복제

리더를 여러 노드로 두면 어떻게 될까? 

모든 쓰기는 로컬에서 처리, 비동기식으로 상대 데이터 센터로 복제. 각 사용자는 특정 데이터 센터로만 읽기 질의를 할 것이니까 괜찮다. 

다만 동일한 데이터를 다른 두개의 데이터 센터에서 동시에 변경하면, 쓰기 충돌 해소가 필요하다.

A라는 제목을 바꾼다고 했을 때 A->B, A->C 누구의 요청을 받아들여야 할까? 단일 리더였으면 두번째걸 차단하겠지만, 다중 리더설정에서는 두 쓰기가 모두 성공한다.  

게다가 자동 증가 키, integrity constraints(PK, unique, FK 등) 같은 문제를 생각해보면 문제될 소지가 있다.

### 협업 편집
구글 독스 같은걸 여러 사람이 편집할 수 있게 한다. 리더가 여러명인 사례같은거야. 로컬 복제 서버에 적용하고, 다른 사용자의 서버에 비동기식으로 복제한다. 

단일 키 입력과 같이 변경 단위를 매우 작게 하면, 해당 키를 동시에 편집하지 않는 이상은 항상 편집이 가능하다. (락이 안걸리니까)

### 충돌 대응

A-> B A->C 어떤 요청 수행되었는지는 비동기 충돌 감지로만 알 수 있다. 

특정 레코드는 모든 쓰기가 동일한 리더에 의해서만 이뤄질 수 있도록 하는게 충돌 회피 방법이다.

Last Write Wins 처럼 타임스탬프를 쓰면 최종 쓰기 승리 방식을 쓸 수도 있다. 또는 명시적 데이터 구조에 충돌을 기록하게 해서, 사용자가 선택하게 한다. 

아무튼 어플리케이션 코드에서 대응해주는게 좋다. 충돌 핸들러 같은걸 정의할 수 있다.

## 리더 없는 복제 
모든 복제서버가 클라이언트로부터 쓰기를 직접 받을 수 있게 허용 

리악, 카산드라, 볼드모트 , dynamo 시스템에서 사용됨 

### 만약 서버 하나가 다운됐다 붙으면?

읽기 요청을 여러 노드에 병렬로 전송하고, 버전 숫자로 최신값을 결정받아 사용한다. 

그럼 따라잡는경우는? 

- 읽기 복구
  - 오래된 응답이라면 클라이언트가 새로운 값 기록 하게 함 (만약 레코드가 잘 읽기 요청이 들어오지 않는다면 그냥 데이터가 계속 누락되어있을 것) 
- 안티 엔트로피 처리
  - 백그라운드 프로세스에서 데이터 차이를 계속 찾는다. 
  - Merkle Tree 방식 해시 비교를 해서 찾아낸다. Root Hash만 비교하면 되기 떄문

```
         Root Hash
        /         \
   Hash A         Hash B
   /    \         /    \
 ...    ...     ...    ...
```

이 hash tree는 데이터 변경시마다 조작되도록 수정 C만 변경되면 그 parent들만 변경된다. 

```
            Root'  ← 변경
          /      \
       H1          H2' ← 변경
     /   \       /    \
   A      B     C'      D
```

## 리더없는 복제 - 동시 쓰기 감지

X에 A를 쓰고 B를 쓴다고 해보자. 어떤 노드에는 B가 먼저 쓰이고 어떤 노드는 A가 먼저 쓰일 것이다. 

### 최종 쓰기 승리 LWW (동시 쓰기 버리기) 
- 가장 최신 값(모든 노드중 최신)을 선택
  - 실제론 서버시간 동기화 안되서 실제 시간기준 최신은 아닐지 몰라도 각자 쓰여진 timestamp 기준으로 

### 이전 발생 관계와 동시성 

A->B 쓰여진건 causally dependent . 고로 B가 나중에 발생해야 한다. 

클라이언트가 항상 쓰기전에 읽어서 최신 버전번호를 사용하게 하고, 이 버전에서 읽어들인 값을 포함시켜서 write해서 발생관계를 포함시킬 수 있다. 

합집합 + 제거했다고 나타내는 툼스톤을 남겨야 최종적으로 제거할 수 있다.
